Almost every control loop in production is some flavor of PID. The equation fits on one line:

```
u(t) = K ( e + (1/Ti) integral(e dt) + Td de/dt )
```

The catch, as Åström puts it, is that a faithful implementation of that line will not give you a good controller. The textbook form ignores noise, saturation, and the realities of running on a sampled processor. Here are the four changes that turn the equation into something you would actually trust on hardware.

## 1. Filter the derivative

A pure derivative has transfer function `s`. Its gain rises without bound, so it amplifies high frequency measurement noise far more than the signal you care about. Differentiating a clean sinusoid plus noise scales the noise by its frequency, and that ratio can be arbitrarily large.

The fix is to band-limit the derivative:

```
D = -K Td s / (1 + s Td/N) Y
```

This acts as a true derivative at low frequency, then flattens out. The high frequency gain is capped at `K N`, with `N` typically between 8 and 20. If you need more attenuation, roll off the whole controller with an extra low pass filter. Tie its time constant to `Td/N` for a PID, or `Ti/N` for a PI, so you do not introduce a free parameter you then forget to tune.

## 2. Do not differentiate the setpoint

If proportional and derivative action both see the raw reference, a step setpoint produces a spike, or even an impulse, in the control signal. The actuator does not enjoy that.

Setpoint weighting solves it cleanly:

```
u = K ( b r - y + (1/Ti) integral(e dt) + Td ( c dr/dt - dy/dt ) )
```

Set `c = 0` so the derivative only ever sees the measurement, never the reference. Use `b` below 1 to soften overshoot on setpoint changes. The integral still acts on the true error, so steady state accuracy is untouched. This is a two degree of freedom controller hiding inside a PID: you tune `K`, `Ti`, `Td` for disturbances and robustness first, then shape the setpoint response with `b` and `c` afterward.

## 3. Anti-windup is not optional

Integral action plus a saturating actuator equals windup. When the output hits its limit the loop is effectively open, but the integrator keeps accumulating error. It can grow large, and then the error has to sit negative for a long time just to unwind it. The result is a big overshoot and a sluggish, bang-bang recovery.

The clean fix is back-calculation. When the output saturates, you recompute the integral so the total output sits exactly at the limit, and you do it gradually with a tracking time constant `Tt`. A common rule of thumb is `Tt = sqrt(Ti Td)`, kept larger than `Td` and smaller than `Ti`.

One practical note worth more than the theory: a fixed integral clamp is a poor substitute. A clamp does not know what the proportional, derivative, and feedforward terms are contributing to the actuator budget, so it either trips too early or lets the real output run past the limit. Back-calculation reacts to the actual saturated output, which is the quantity that matters. In my experience the difference shows up most under heavy load, exactly when you need recovery to be crisp.

## 4. Mind the implementation form

Two controllers can be algebraically equivalent and still behave differently when you tune them. A few habits from Trumper's MIT 2.140 notes are worth keeping:

- Put the loop gain `K` out in front, not buried inside one term, so a gain change scales every term together and does not drag your zeros around.
- Prefer a series lag/lead structure. Its zeros sit in known, easily located positions, and every lead zero gets a pole to go with it so the high frequency gain stays bounded.
- Be wary of the naive parallel form. It can hand you complex controller zeros for some parameter choices, and the academic version has more zeros than poles. It is convenient for computing closed loop poles in a homework problem and not much else.

## 5. Discrete-time PID algorithm pseudocode

```
# --- precompute, only when parameters change ---
bi = K * h / Ti              # integral gain
ad = Td / (Td + N*h)         # derivative state decay  (eq 6.17)
bd = K * N * Td / (Td + N*h)  # derivative gain
ao = h / Tt                  # anti-windup (tracking) gain, Tt = sqrt(Ti*Td)

# --- persistent state ---
I    = 0      # integral term
D    = 0      # filtered derivative term
yold = 0      # previous measurement

# --- run once per sampling period h ---
loop:
    r = read_setpoint()
    y = read_measurement()

    P = K * (b*r - y)                 # proportional, setpoint-weighted
    D = ad*D - bd*(y - yold)          # derivative, filtered, on -y only
    v = P + I + D                     # unsaturated controller output

    u = saturate(v, u_low, u_high)    # actuator (or actuator model)
    write_output(u)

    I = I + bi*(r - y) + ao*(u - v)   # integrate error + back-calculation
    yold = y
```
## Tuning is the last ten percent

Once the structure is right, tuning is the easy part. Ziegler-Nichols gives you a ballpark, nothing more. The original rules targeted quarter amplitude decay, which corresponds to a damping ratio around 0.2 and leaves the loop poorly damped and short on robustness. Modern rules that constrain the maximum sensitivity, say `Ms` near 1.4, give you a far better starting point for the same effort.

The summary I keep coming back to: PID suits processes with roughly monotone step responses and modest requirements. PI alone gives no phase advance, so it stalls on anything with 180 degrees of lag or more, like a double integrator. Derivative action buys you up to about 50 degrees of phase lead, which is why it earns its place on the harder, delay-light plants. Know which regime you are in before you reach for `Td`.

None of this is exotic. It is the difference between an equation and a controller.
